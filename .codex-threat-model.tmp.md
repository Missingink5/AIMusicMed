# AIMusicMed Repository Threat Model

## Overview

AIMusicMed is an invite-only, publicly reachable web application that turns private emotional conversations and user-selected settings into generated music-meditation audio. The deployed product is a single-host Docker Compose stack: Caddy is the only intended public ingress; a TypeScript/React frontend and FastAPI API run behind it; a private worker performs AI-provider calls and audio processing; SQLite and private media share an application data volume; and a Restic-based backup service plus a host-side systemd request runner provide backup and restore operations. The repository also contains a local CLI generation workflow, but the web stack is the primary production surface.

The principal protected assets are:

- account identities, invitation state, passwords, session tokens, verification challenges, password-reset grants, and administrator step-up grants;
- highly sensitive conversation text, inferred emotions, meditation plans, job history, and generated guidance text;
- uploaded private music, voice-cloning source recordings and previews, generated WAV/MP3/TXT works, ownership metadata, consent declarations, and deletion/retention state;
- platform DeepSeek, MiniMax, ElevenLabs, Tencent SES credentials, user-supplied BYOK credentials, the API-worker bearer token, the Fernet key, the Restic passphrase, and backup archives;
- administrator authority, audit records, user quotas, global concurrency and provider-spend controls, and the integrity of worker job state;
- availability of the small single-host deployment, its limited disk/memory budget, and recoverable SQLite/media state.

The model covers production/runtime code in `webapp/`, `web/app/`, the root audio/provider modules, `docker-compose.yml`, and `deploy/`. Tests and developer tooling inform intended controls but are not themselves primary attack surfaces. `作品介绍/`, media binaries, dependency directories, and build outputs are excluded from source review. This is a threat model, not a list of concrete vulnerabilities.

## Threat Model, Trust Boundaries, and Assumptions

### Actors and trust levels

- **Unauthenticated Internet users** can reach Caddy, the frontend, public health/authentication endpoints, and any API route exposed under `/api/v1/`. They may automate requests, supply arbitrary email strings, codes, passwords, headers, and protocol-level input.
- **Invited authenticated users** are partially trusted only for their own tenant. They control conversation text, plan settings, filenames, tags, edit parameters, private API keys, music/voice uploads, job actions, and requested downloads. One invited user must be treated as hostile to every other user's data and provider budget.
- **Administrators** are privileged application users, but their browser, session, and uploaded backup files remain attackable. Sensitive administration actions are expected to require a fresh, purpose-bound email-code grant and to be auditable.
- **The API and worker** trust each other through a shared bearer token over the private Docker network. The worker receives decrypted BYOK credentials, storage paths, selected media, and job specifications, then invokes FFmpeg and external providers. Compromise of either service can affect private application data; worker compromise also exposes transient provider credentials and generated artifacts.
- **The host backup request runner** is a separate, highly privileged boundary. It runs as root with Docker socket access and converts narrowly structured request files into fixed Docker Compose operations. Application code must not gain general command, path, environment, or Docker control through this queue.
- **The operator/deployer** controls environment files, domain/DNS, TLS deployment, Docker/systemd installation, database/backup keys, provider credentials, retention settings, and restoration commands. These are trusted administrative inputs, not attacker-controlled web inputs.
- **Developers and the software supply chain** control source, lockfiles, container bases, Python/npm dependencies, migrations, and deployment scripts. A compromised dependency or image can execute with the privileges of the corresponding service.
- **External processors** include Tencent SES, DeepSeek, MiniMax, and optionally ElevenLabs. They receive authentication messages, user text or derived prompts, voice samples, and/or generated audio according to the selected workflow. Their availability, billing, authentication, content handling, and response integrity are outside the repository's direct control.

### Trust boundaries

1. **Internet to Caddy:** TLS termination, hostname routing, security headers, and the rule that `/api/v1/internal/*` is not publicly forwarded.
2. **Browser to FastAPI:** cookie-authenticated, state-changing requests and streamed/downloaded content. The browser may render untrusted user/provider text; the API must enforce authorization independently of UI state.
3. **User to tenant storage/database:** every conversation, message, plan, job, work, private music item, voice clone, notification, and BYOK secret must remain bound to its owner. Public music is administrator-managed and intentionally crosses user boundaries only as read-only shared content.
4. **API to worker:** bearer-token-authenticated job and voice-clone claim/progress/completion calls. Payloads include sensitive paths and decrypted credentials, so this is a privileged internal protocol.
5. **Worker to local tools/files:** FFmpeg/audio decoders and filesystem operations consume attacker-supplied or provider-returned media. Media is untrusted structured binary data even when its extension and magic bytes are checked.
6. **Application to external providers:** outbound HTTPS carries private text, prompts, voice samples, provider keys, and billable requests. Provider responses are untrusted JSON/audio and must not be allowed to control local paths, commands, or authorization decisions.
7. **API to backup request filesystem to root runner:** the API may enqueue only allowlisted operations and package identifiers. The runner's Docker socket and root privileges must never become a general execution primitive.
8. **Live data to Restic/offline packages:** backup confidentiality depends on a separate passphrase; backup integrity and restore safety depend on checksums, repository verification, explicit snapshot selection, safe archive extraction, and non-overwriting drills before live restoration.

### Security invariants

- Only the configured administrator and explicitly invited email accounts can activate and log in; authentication responses should not disclose whether arbitrary addresses are allowlisted.
- Passwords use a memory-hard password hash. Sessions, verification codes, reset tokens, worker tokens, and step-up grants must remain unpredictable, hashed where persisted, expiration-bound, purpose-bound, and revocable.
- Cookie authentication must remain `HttpOnly`, `Secure` in production, and `SameSite=Strict`; Caddy must remain the sole public ingress and internal worker endpoints must remain unreachable from the Internet.
- Authorization is server-side and object-specific. Possession or guessing of another object's ID must never grant access to conversations, messages, plans, jobs, works, uploads, voice clones, notifications, API credentials, or downloads.
- Administrator role checks and fresh one-time action grants must precede sensitive quota, user-status, public-library, voice, work, job, backup, and restore operations. Audit records must not contain plaintext secrets or private conversation content.
- User BYOK credentials remain encrypted at rest, are decrypted only for the requesting user's job, and never appear in API responses, logs, manifests, analytics, or downloadable artifacts. Platform and SES credentials, the Fernet key, worker token, and backup passphrase remain deployment secrets.
- All attacker-influenced paths stay beneath the intended storage or backup root. Uploaded filenames, archive entries, database paths, provider identifiers, and job metadata must not cause traversal, symlink escape, arbitrary overwrite, or command construction.
- Uploaded and provider-returned media is bounded by type, byte size, duration, quota, and disk-pressure policy before expensive processing. Audio tooling receives argument arrays or fixed filter graphs rather than shell-interpreted user strings.
- Daily platform generation limits, per-user single-job rules, global concurrency, clone quotas, upload quotas, timeouts, retry bounds, and disk/memory guards prevent one user or provider failure from creating uncontrolled cost or exhausting the host.
- Private emotional text and voice data are sent only to processors required for the selected workflow and are retained/deleted according to product policy. Anonymous analytics must remain opt-out-aware, allowlisted, low-dimensional, and separated from message, identity, path, token, and credential data.
- Backup creation, verification, import, and restore preserve confidentiality and integrity. Destructive live restore requires explicit operator confirmation and an immutable snapshot ID, captures the current state first, and cannot be triggered as arbitrary host execution.

### Assumptions and out-of-scope stories

- Production DNS and TLS keys, the Ubuntu/Docker host, provider accounts, and the operator's external password manager are assumed not already compromised. A fully privileged host or cloud-account compromise can bypass application controls and is an operational incident rather than an application authorization bypass.
- Caddy is assumed to be the only exposed service; ports 3000/8000, SQLite, Docker, and the backup repository are not Internet-accessible. Risk changes materially if that deployment assumption is violated.
- The product is intended for fewer than five invited users and does not provide public sharing. This lowers broad anonymous exposure but does not reduce the required isolation between invited users, the sensitivity of voice/emotional data, or the impact of administrator compromise.
- AIMusicMed is a wellness/meditation product, not a clinical diagnostic, crisis-response, or medical-treatment system. Incorrect therapeutic advice is primarily a safety/product-governance concern unless security compromise causes deceptive content, data exposure, or denial of explicitly promised safeguards.
- Provider-side retention, model behavior, account billing disputes, and provider infrastructure compromise are not fixed by this repository, but the application is responsible for minimizing data sent, authenticating requests, bounding retries/cost, and safely parsing responses.

## Attack Surface, Mitigations, and Attacker Stories

### Public web, authentication, and sessions

Relevant surfaces include the Caddy routing configuration, FastAPI authentication endpoints, React login UI, SES delivery, session cookies, password setup/reset, invitation administration, and code/password rate limits. Realistic attacker stories include credential stuffing, six-digit-code guessing, email flooding, account enumeration, session theft through browser compromise, cross-site state changes, and attempts to reach internal worker routes.

Repository controls include generic code-request responses, HMAC-hashed email/IP throttle keys, send/verify attempt limits, 15-minute one-time codes, Argon2 password hashing, dummy password verification for unknown accounts, lockouts, hashed random session/reset tokens, strict secure cookies, server-side role checks, forced initial administrator password setup, purpose-bound administrator grants, and Caddy denial of public internal routes. These controls depend on production settings keeping development code disclosure disabled and secure cookies enabled.

### Tenant APIs, conversations, streaming, jobs, and downloads

Conversation text and emotional context are attacker-controlled and privacy-sensitive. Plans, retries, cancellations, streaming assistant output, job polling, favorites, and downloads all cross the user-to-tenant boundary. Relevant failure classes are missing object ownership checks, mass assignment of plan/job state, unsafe rendering of stored/provider text, leakage through errors or filenames, stale authorization, and state-machine races between user cancellation and worker completion.

Existing design signals include dependency-based authentication, repeated `user_id`/conversation ownership predicates, parameterized SQLite queries, Pydantic bounded fields and literals, explicit job states, one-running-job checks, platform daily quotas, and safe storage-root resolution before downloads. The frontend uses same-origin credentialed requests; any HTML-rendering or service-worker caching change must preserve the rule that authenticated/private API responses are not cached into a cross-user or public cache.

### BYOK secrets and external AI/SES integrations

Users supply DeepSeek and MiniMax keys and may supply ElevenLabs keys. The platform also has shared provider and SES credentials. Attacker stories include stealing another user's key, causing keys to enter logs or manifests, confusing platform and BYOK billing, replaying billable requests, oversized or malformed provider responses, and prompt manipulation that changes content or induces excessive output.

Controls include Fernet encryption at rest, omission of key values from normal responses, per-user credential lookup, explicit credential mode, bounded provider prompts/durations, request timeouts, limited retry/fallback behavior, validation of returned JSON/audio containers, and a private worker channel. Because the worker receives plaintext keys transiently, worker authorization, logging discipline, error sanitization, and isolation are high-value controls. Provider content must be treated as data, never as instructions to the application runtime.

### Uploads, voice cloning, audio processing, and private storage

Authenticated users can upload music and voice recordings; administrators can upload public music and encrypted backup packages. Inputs include arbitrary bytes, extensions, MIME claims, metadata, emotion tags, edit parameters, and consent assertions. Relevant classes are parser/decoder exploitation, decompression or resource exhaustion, traversal and overwrite, quota bypass, cross-tenant media access, unauthorized voice cloning, and retention/deletion divergence between local and provider copies.

The repository has streaming byte limits, per-user storage/count/duration quotas, disk-pressure stop thresholds, extension and signature checks, resolved storage paths, atomic temporary files, authorization declarations with content hashes, clone-attempt/slot limits, provider deletion actions, and non-shell FFmpeg invocation. These controls should be preserved across every preview, edit, retry, and cleanup path. Consent records support auditability but do not by themselves prove that an uploaded voice or music asset was lawfully authorized.

### Administrator and operations console

Administrator surfaces manage users, quotas, public music, cloned voices, jobs, works, analytics, notifications, audits, system state, and backups. A stolen administrator session or persistent browser injection can therefore have broad impact. The intended defense is layered: administrator role checks, mandatory initial password, fresh emailed step-up codes for enumerated sensitive actions, single-use purpose-bound grants, hashed-IP audit events, and notification/alerting.

Read-only administrator views should continue to avoid exposing conversation bodies and plaintext secrets. New administrative mutations should default to explicit step-up authorization, narrow inputs, atomic state transitions, and auditable outcomes rather than relying on a hidden UI button.

### Worker, filesystem, and provider state machines

The worker claims one job globally, receives trusted API-created paths and selected asset identifiers, invokes the generation pipeline and FFmpeg, calls external providers, writes artifacts through partial files, and reports progress/completion. Relevant failure modes include forging worker callbacks, claiming another user's credentials or media, path escape, command injection, corrupted media, duplicate paid operations, stale-job resurrection, and disk/memory exhaustion.

The API-worker bearer token, private Docker network, Caddy internal-route denial, global concurrency, heartbeat/stale-job handling, fixed subprocess argument arrays, provider response validation, and partial-file replacement reduce these risks. The worker token is effectively a service credential with broad job authority and must be strong, rotated on exposure, excluded from logs, and inaccessible to the browser.

### Backup, restore, and host automation

Backups contain the full SQLite database, emotional conversations, voice recordings, generated works, encrypted BYOK fields, and authorization/audit records. The API writes structured requests into a bounded directory; a root systemd runner with Docker socket access maps allowlisted actions to fixed backup-container commands. This is the repository's most privileged operational boundary.

Controls include restrictive systemd settings and filesystem permissions, fixed action dispatch, safe package-name handling, encrypted Restic repositories, explicit hexadecimal snapshot IDs for import/restore, archive-entry validation, SHA-256 sidecars, full Restic data checks, SQLite integrity checks, pre-restore snapshots, non-overwriting drill/item targets, atomic request/result files, and service stop/start sequencing. Attacker-controlled backup archives remain hostile even when uploaded by an administrator. Restore code must never accept archive paths, shell fragments, Compose options, host paths, or environment names from a web request beyond narrowly validated identifiers.

### Availability, retention, privacy, and supply chain

The deployment has a 2 GB host and finite disk, making long media jobs, uploads, backup pruning, and provider retries realistic denial-of-service and cost surfaces. Memory limits, global single-task concurrency, byte/count quotas, log rotation, disk hysteresis, backup repository caps, and health checks are material mitigations. Availability controls must fail closed for new uploads/generation without corrupting existing user data.

Conversation/audio deletion and 30-day work expiration must remain consistent across SQLite, live storage, provider voice state, caches, and retained backup snapshots. Anonymous metrics should not become identifying in a user population smaller than five; aggregate count suppression and default-disabled collection are appropriate assumptions for this deployment. Dependency lockfiles, pinned direct dependencies, non-root API/web containers, narrow Docker build contexts, and routine rebuilds reduce supply-chain risk, while the root backup runner and media parsers deserve especially conservative updates.

## Severity Calibration (Critical, High, Medium, Low)

### Critical

Critical impact requires repository-wide compromise or loss of the most sensitive assets, for example:

- unauthenticated remote code execution in the public API/media path, or a web-to-backup-queue escape that yields root/Docker control;
- authentication or authorization failure enabling bulk export of all conversations, voice recordings, BYOK/platform credentials, or encrypted backups together with their key material;
- compromise of the worker service credential that can be escalated to arbitrary host execution or broad secret extraction;
- live backup restoration or destructive filesystem control triggerable by an unauthenticated or ordinary user.

The same bug class is lower if it is confined to a developer-only CLI with no production path and no realistic attacker-controlled input.

### High

High impact includes substantial account, tenant, privacy, or privileged-operation compromise without full host takeover, for example:

- bypassing invitation/login protections, taking over an administrator or invited account, or reusing a step-up grant for a different sensitive action;
- cross-user access to conversation text, private music, voice samples, generated works, or another user's BYOK credentials;
- persistent script execution in an administrator's browser that can perform privileged application actions;
- arbitrary read/write inside application or backup data volumes, forged worker completion, unauthorized voice cloning/deletion, or unbounded provider spending across accounts;
- restore/import integrity failure capable of replacing live application state with attacker-controlled but non-host-executing data.

### Medium

Medium impact is meaningful but bounded to one user, a limited data class, or recoverable service degradation, for example:

- rate-limit bypass enabling targeted email abuse or code guessing without a demonstrated account takeover;
- a CSRF-like state change limited by strict cookies and not reaching administrator or credential operations;
- metadata leakage, limited stored-content exposure, or a one-user quota/accounting bypass;
- uploads or generation requests that exhaust a container, temporarily fill guarded storage, or cause bounded duplicate provider charges;
- retention, cache, analytics, or deletion inconsistencies that expose limited non-secret data but not raw conversations, voice recordings, or credentials.

Risk rises when the condition is remotely repeatable against all invited users or can chain into account/administrator compromise.

### Low

Low impact includes issues with minimal confidentiality, integrity, or availability consequence, for example:

- non-sensitive version, health, or operational metadata disclosure that does not aid a practical exploit;
- minor error-message, cache-header, UI-only authorization-display, or audit-format inconsistencies when server-side controls remain intact;
- low-rate resource inefficiency that stays within existing quotas and does not materially affect availability or cost;
- weaknesses confined to tests, documentation, examples, local developer scripts, excluded media, or build outputs with no production execution path.

Pure product-quality concerns, meditation-content preferences, and claims about clinical efficacy are outside security severity unless an attacker can exploit them to violate a stated security, privacy, integrity, or availability invariant.
